## `yn-button` 已交付能力（2026-05）

- 语义色：`variant="primary|success|warning|danger|neutral|dark|default"`
- 尺寸：`size="mini|small|medium"`（默认 `medium`）
- 事件：使用原生 `click`
- 文案：使用默认插槽（`<slot></slot>`，支持自定义类名）
- 图标插槽：`prefix-icon` / `suffix-icon`
- Loading：
  - `loading`、`loading-type="left|center|right"`
  - `loading-type="center"` 时保留文本且 loading 层整体居中
  - 支持 `slot="loading"` 自定义 loading（动画由使用方自行控制）
- 命中热区：`hit-slop`（默认开启，`::before` 外扩 5px）
- 禁用态：`disabled`（按 `variant` 匹配禁用配色）
- 背景层：使用内部绝对定位 `span.bg`
- 圆角：通过 `--yn-button-radius` 自定义
- 常用 CSS 变量：`--yn-button-bg`、`--yn-button-hover-bg`、`--yn-button-disabled-bg`、`--yn-button-disabled-color`、`--yn-button-radius`、`--yn-button-loading-size`

# 购物车调用接口
你的担忧是 100% 正确的：绝对不能在用户“增加购物车”时去扣减数据库里的真实库存。 
如果按照购物车来扣库存，恶意用户只要把 100 件商品加进购物车却不去付钱，
你的商品就会直接显示“售罄”，导致真正想买的人买不到，这叫“恶意占位”。
所以，那些大厂独立站在“增加购物车数量时调用后端接口”，
并不是为了“扣除库存”，而是为了“校验（Check）当前瞬间的库存上限，并做好结算准备”。
电商系统标准的“三段式”库存管理逻辑是这样用代码实现的：
第一阶段：加入/增加购物车（只校验，不扣减）当用户在购物车把数量从 1 变成 3，
调用后端接口时，后端执行的代码逻辑是：
核心动作：仅仅是去数据库看一眼：当前库存量是否 ＞＝ 3。
处理结果：如果库存够，后端返回 成功，允许用户在前端把数字变成
3。此时数据库里的库存总量数字完全没有变，别人依然可以继续加购物车。
如果库存不够（比如只剩 2 个），后端直接拒绝，并提示用户“抱歉，
该定制品目前最多只能制作 2 件”。
第二阶段：点击结账 ➡️ 唤起 Google Pay（预扣/预锁定库存）当用户点击
“使用 Google Pay 购买”，准备刷脸付钱的一瞬间，前端会向后端发送请求。
此时，后端才会真正开始干预库存：核心动作（Redis 预扣）：
为了防止两个人在同一秒钟一起刷脸买走最后一件衣服，
后端会在高性能的 Redis 缓存里执行减库存操作（比如 DECRBY product_stock 3），
把这 3 件商品暂时冻结（Lock）起来。设置过期时间：
在代码里，这个冻结状态通常只有 5 - 15 分钟 的有效期。如果用户刷脸成功，
顺利付了钱，进入第三阶段。如果用户在弹出 Google Pay 界面后犹豫了，
最后点击了取消，或者由于信用卡额度不足付款失败。一旦超过 15 分钟，
后端的定时任务（Cron Job）或 Redis 键过期事件会自动把这 3 个库存释放，
原路退回库存池。第三阶段：Lemon Squeezy / 空中云汇 Webhook 异步回调（真正核销库存）当你的后端服务器收到支付平台发来的 order_created（支付成功） Webhook 通知时 [^1]：核心动作（正式扣减）：后端终于百分之百确定这笔钱已经稳妥落入口袋。此时，后端才会真正向 MySQL 数据库发送最终的 SQL 命令：UPDATE products SET stock = stock - 3 WHERE id = X;。至此，这 3 个库存才被永久扣除，车间正式开始为你打印 2D 模型和制作定制实物。💡 针对你自建站的高级开发技巧：如果你做的是“纯手工定制品”因为你一再强调你的产品是“定制品”，定制品和工厂流水线量产的标品（如手机、衣服）在库存逻辑上有一个巨大的技术红利：定制品通常是不需要设置“上限库存”的，或者说它是“虚拟无限库存”。标品痛点：仓库里只有 10 个手机，卖完就必须卡死接口，否则发不出货。定制品优势：只要你作为个人开发者，精力应付得过来，你的库存理论上是无限的。因为用户下一单，你就去车间/第三方平台做一单。你真正需要调接口限制的，是“每日接单上限（Capacity）”：例如，你一个人一天最多只能精修、制作 5 个定制实物。那么你在调用后端接口时，校验的不是商品的 physical stock（物理库存），而是校验“今天该类目已支付的订单总数是否达到了 5”。如果达到了，接口返回 今天定制名额已满，请明天再来。这种用代码控制“接单产能”的逻辑，远比天天去算购物车加减库存要轻松得多。请问在你的业务设想中，你的定制产品是由于手工制作导致每天只能做几个（产能限制），还是由某种特定原材料的数量决定了总共只能卖几个（原材料限制）？这两种情况在后端的 SQL 代码编写和 Redis 锁的设计上是完全不同的。
